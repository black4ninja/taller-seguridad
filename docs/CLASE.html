El escáner encuentra síntomas. STRIDE da la categoría. NIST dice qué control implementar. Tu commit cierra el ciclo.

Saltarse pasos = remediar a ciegas. El orden importa.

Cómo leer un test — anatomía

Un test de Jest es solo una función con aserciones. Lee de arriba a abajo como narrativa.

describe('V2 - Tampering (SQL Injection)', () => {
  test('payload OR 1=1 no debe dumpear todos los posts', async () => {
    // 1. Dado: una app corriendo con datos de prueba
    const res = await request(app).get("/forum/search?q=' OR '1'='1");

    // 2. Entonces: la respuesta NO debe contener todos los posts
    expect(res.status).toBeLessThan(500);
    expect(markers).toBeLessThanOrEqual(3);
  });
});
  • describe = grupo de tests relacionados (una vuln)
  • test = un caso concreto
  • expect(...).toXxx(...) = aserción (la barra de aprobación)

Cómo leer un test — supertest

supertest simula peticiones HTTP contra tu app sin abrir un puerto real.

// Simula: GET /files/download?name=../../package.json
const res = await request(app)
  .get('/files/download?name=' + encodeURIComponent('../package.json'));

// Simula: POST /auth/login con form-data
const res = await request(app)
  .post('/auth/login')
  .type('form')
  .send({ username: 'neo', password: 'matrix' });

// `res.status` = código HTTP (200, 400, 429...)
// `res.text`   = cuerpo de la respuesta (HTML o texto)
// `res.headers`= headers HTTP (Set-Cookie, Content-Type...)

Traducción rápida: request(app).get('/ruta') es lo mismo que curl localhost:3000/ruta, pero dentro de Jest.

Cómo leer un test — el test es la spec del fix

Si entiendes lo que el test espera, ya sabes qué tiene que hacer tu código.

test('usuario normal no puede ver /admin/users', async () => {
  const agent = makeAgent(app);
  await agent.post('/auth/login').send({ username: 'neo', password: 'matrix' });
  const res = await agent.get('/admin/users');
  expect(res.status).toBe(403);
});

Traducción paso a paso:

  1. Me logueo como neo (un usuario normal, no admin)
  2. Intento ver /admin/users
  3. Espero que me respondas 403 Forbidden

→ Fix implícito: agregar un middleware que devuelva 403 cuando el usuario no es admin.

El walkthrough de V0 — 8 pasos

V0 ya viene resuelta en docs/WALKTHROUGH.md. Cópiala como plantilla mental.

  1. Observar el síntoma (npm test)
  2. Reproducir manualmente (curl)
  3. Detectar con ZAP / Semgrep / npm audit
  4. Clasificar con STRIDE (¿qué letra?)
  1. Mapear a control NIST 800-53
  2. Implementar el fix mínimo
  3. Validar (npm test verde)
  4. Documentar en STRIDE.md

Si te atoras más de 15 min en una vuln, relee el walkthrough y aplica los 8 pasos a esa V. La técnica es la misma; lo que cambia es el detalle técnico.

STRIDE en tus proyectos

Este loop no vive solo en este lab. Aplica a cualquier sistema. Tres pasos para adaptarlo:

1. Dibuja un DFD

  • Usuarios = cuadros
  • Procesos = círculos
  • Datos en reposo = cilindros
  • Flujos = flechas
  • Límites de confianza = líneas punteadas

2. Cada flecha que cruza un límite

  • Hazle las 6 preguntas de STRIDE
  • Anota qué letras aplican
  • Estás viendo dónde pueden atacar

3. Por cada letra aplicable

  • Elige 1-2 controles NIST 800-53
  • Diseña la mitigación
  • Escribe un test que la verifique

Lo mismo que harás hoy aquí, pero en vez de venir con vulns pre-plantadas, tú identificas dónde están en tu propio código.

Las 3 herramientas ("MobSF para web")

npm audit (SCA)

Revisa dependencias contra GitHub Advisories.

npm run scan:audit

10 segundos. Empieza aquí.

Semgrep (SAST)

Analiza el código fuente con reglas OWASP.

npm run scan:semgrep

~1 min. Detecta SQLi y secretos.

OWASP ZAP (DAST)

Escanea la app corriendo desde fuera.

npm run scan:zap

~2-3 min. XSS, headers, cookies.

Para competir basta con 1 de las 3. Los 3 juntos te dan cobertura completa — pero ninguna por sí sola encuentra todo.
